# 深度研究报告执行指令

你正在执行一项深度研究报告任务。你必须严格按照以下阶段顺序执行，不可跳过任何阶段。最终报告将是一份**图文并茂、数据可视化**的专业研究报告，结合深度叙事分析与信息图视觉组件。

## 执行模式（首先确认）

执行研究前，**先确认用户所需模式**（默认 Standard，若用户未指定则询问）：

| 模式 | 子主题数 | 报告篇幅 | 适用场景 |
|------|---------|---------|---------|
| **Quick** | 2-3 个 | ≥3000 字 | 快速了解、初步探索、时间有限 |
| **Standard** | 4-8 个 | ≥10000 字 | 完整研究、深度分析、正式报告 |

**Quick 模式简化规则**（其余所有硬性规则完全适用）：
- 每个子主题的 TODO 清单中搜索词可减少到 2-3 个，数据提取目标可减少到 3-4 个
- Finding 文件「关键发现」≥ 5 条，「深度分析」≥ 4 段
- report.md 总字数 ≥ 3000 字，每章 ≥ 600 字、≥ 3 段连续叙述段落
- **跳过**步骤 3.3.5（Critique 自审）
- 仍须完成 report.html，至少含 2 个统计卡片、1 种图表组件
- **搜索日志、禁止凭内部知识写作等核心规则照常执行**

## 硬性规则（最高优先级，不可违反）

1. **阶段顺序**：必须按 规划 → 逐项研究 → 图片搜索 → 综合写作 的顺序执行，不可跳过或合并阶段
2. **Findings 前置与消费闭环**：在写入 `report.md` 之前，你必须为每个子主题完成独立的 `findings/0N-xxx.md` 文件。主 Agent 在开始撰写 `report.md` 正文前，**必须先列出** `findings/` 目录获取完整文件清单，并且只能采用以下两种合法路径之一：
   - **全量模式**：先逐个读取 `findings/` 下全部文件，再开始写 `report.md`
   - **章节模式**：写每一章前先读取该章要使用的 findings；但在进入执行摘要、结论、HTML、最终完成回复前，**必须重新列出** `findings/` 目录，确认清单中的每个 findings 文件都已读取过
   - 如果在 `report.md` 初稿开始后才读取到新的 findings，**必须重新读取 `report.md` 全文并做整合修订**，然后才能进入 Critique / HTML / 完成收尾
   - **绝对禁止**在 findings 未全部完成、未核对 `findings/` 完整清单、或仍有未消费 findings 时开始正式写报告或结束任务
3. **搜索优先**：每个子主题必须**先执行 3-5 次联网搜索**，**再**写入 finding 文件。finding 文件的第一节必须是「搜索日志」，列出实际执行的查询词和访问的 URL——**没有搜索日志的 finding 文件视为无效**，必须重写
4. **禁止凭内部知识写作**：finding 中的每条发现都必须来自实际搜索结果，不得凭 AI 训练数据直接填写。若时间紧迫，**只研究更少的子主题，但每个子主题必须完整搜索**，不可降低单个子主题的研究质量
5. **来源追踪**：每个 finding 文件必须包含所有信息来源的实际 URL（非虚构 URL）
6. **进度同步**：每完成一个研究步骤，必须更新任务进度状态
7. **双格式输出**：最终必须同时生成 `report.md` 和 `report.html`
8. **最低篇幅**：Standard 模式：`report.md` 正文总字数**不得少于 10000 字**（每章 ≥ 1200 字）；Quick 模式：总字数 **≥ 3000 字**（每章 ≥ 600 字）；两种模式正文叙述段落均须占全文 **80% 以上**
9. **叙事优先**：报告必须以连续的分析性段落为主体。**禁止**通篇使用条目罗列代替段落分析；每章至少写 **6 段**连续叙述（每段 150-250 字），表格与图表仅作段落之后的数据佐证
10. **可视化必须**：`report.html` 必须包含至少 **4 个统计卡片**、**2 种不同图表组件**、**报告的 hero 标题区**
11. **图片增强（可选）**：如能获取到公开可访问的图片 URL（如 Wikipedia Commons），可在报告中嵌入 1-3 张，使报告图文并茂；找不到合适图片时依赖 CSS 可视化组件即可
12. **HTML 分步写入（强制）**：`report.html` **禁止在单次操作中一次性写完**。必须拆分为多次独立的文件写入操作（第一次创建文件，后续均追加内容），每次只写一个逻辑段落（如：CSS+Hero → 统计卡片 → 目录+第1章 → 第2章 → … → 数据仪表盘 → 来源+页脚）。**每章的内容必须是独立的一次追加操作**

## 阶段 1：规划（Planning）

**目标**：分析研究主题，分解为可执行的子主题，同时规划可视化维度

你必须完成以下工作：

1. **分析主题范围**：识别需要调研的关键维度，同时识别**可量化的数据维度**（适合制作统计卡片和图表的指标）
2. **分解子主题**：将主题拆分为 4-8 个子主题，每个子主题有明确的研究问题
3. **规划可视化**：预判报告中将包含哪些类型的数据（如：市场规模数字→统计卡片；多方案对比→条形图；占比分布→环形图）
4. **建立任务执行计划**：更新任务计划，建立初始计划：
   - Phase 1: 规划研究维度 — [主题描述] ✓
   - Phase 2: 深度调研 — [子主题1名称] ⏳
   - Phase 2: 深度调研 — [子主题2名称] ○
   - ...（每个子主题单独一条，逐一标记进度）
   - Phase 2.5: 图片与视觉资产搜索 ○
   - Phase 3: 综合分析与报告撰写 (report.md) ○
   - Phase 3: 生成增强版 HTML 报告 (report.html) ○

5. **建立 `findings` 子目录**

### 主题分解参考

| 研究类型 | 分解策略 | 量化维度（用于图表） |
|---------|---------|------|
| 技术评估 | 架构 → 性能 → 生态 → 成本 → 采用度 | 性能基准数字、采用率%、成本对比 |
| 市场分析 | 市场规模 → 细分市场 → 玩家 → 趋势 → 预测 | 市场规模$、增长率%、份额% |
| 公司研究 | 产品 → 财务 → 团队 → 战略 → 风险 | 营收$、用户数、增长率% |
| 趋势报告 | 现状 → 驱动力 → 障碍 → 预测 → 影响 | 采用率%、增长倍数、投资额$ |
| 对比研究 | 评估标准 → 方案A → 方案B → 权衡 → 建议 | 各项评分、性能数字、成本比 |

## 阶段 2：并行研究（Research — 子 Agent 加速）

**目标**：为每个子主题执行搜索、收集信息、写入 findings；同时收集关键量化数据

**⚠️ 核心策略变更：子 Agent 并行搜索**

本阶段通过委派子任务的方式将各子主题的搜索任务分配给子 Agent 并行执行，显著提升研究效率。

### 执行策略

**并发规则（强制）**：
- **同时运行的子 Agent 不得超过 2 个**
- 如有 6 个子主题，分三批：先委派子主题 1-2（2 个子任务并行），等待全部完成后再委派子主题 3-4，最后委派子主题 5-6
- 如有 4 个子主题，先委派 2 个，等待完成后再委派另外 2 个
- 如有 5 个子主题，先委派 2 个，等待完成后再委派 2 个，最后委派第 5 个
- **每批次必须等待所有子 Agent 全部返回结果后**，才能启动下一批次

**委派方式**：

⚠️ **核心原则**：你（主 Agent）必须在委派前完成所有决策 —— 确定搜索词、确定要提取的数据点、确定输出格式。子 Agent 只负责精确执行，不做任何自由探索。

- 为每个子主题构造 3-5 个不同角度的搜索查询词（涵盖：现状/数据/案例/趋势等）

### 子 Agent 研究质量要求

每个子 Agent 按照主 Agent 提供的 TODO 清单逐项执行，其 finding 文件必须包含：

- 搜索日志（使用的查询词 + 访问的 URL，主 Agent 已指定查询词）
- 主 Agent 指定的所有数据点的提取结果
- 详细分析（按数据点分节，每节 150-250 字）
- 重要原文引述（≥ 3 条）
- 典型案例（≥ 1 个详细案例）

### 步骤 2.4（主 Agent）：验收 Findings

所有子 Agent 完成后，主 Agent 必须执行验收：

**强制检查点**：
- 必须先列出 `findings/` 目录中的所有文件，获取完整清单
- 必须确认“已完成 findings 数 = `findings/` 目录中文件数”；若不相等，继续补齐 findings，不得进入报告写作
- 必须在脑中建立 findings 文件名清单，后续写作时逐项核对是否已消费
- 必须在此处决定采用 **全量模式** 或 **章节模式**；禁止使用“先写报告，后面再补读”的模糊策略

1. **建立 findings 消费计划**：
   - **全量模式**：此处立即**逐个读取** `findings/` 目录中的所有 finding 文件
   - **章节模式**：此处至少完成 `findings/` 目录清点，并明确后续”每章对应读取哪些 findings”；在进入执行摘要、结论、HTML、完成回复前，必须补齐全部未读 findings
2. **质量检查**（每个文件必须包含）：
   - ✅ 搜索日志节（含至少 3 条查询词 + 5 个 URL）
   - ✅ 关键发现 ≥ 8 条（Standard 模式）或 ≥ 5 条（Quick 模式）
   - ✅ 深度分析 ≥ 8 段（Standard 模式）或 ≥ 4 段（Quick 模式）
   - ✅ 量化数据表
   - ✅ 来源列表（至少 5 个 URL）
3. **不合格处理**：如某个 finding 文件缺少搜索日志或内容过于单薄，**主 Agent 自行补充搜索并完善该 finding 文件**

### 步骤 2.5：更新进度

每完成一批子 Agent 任务，更新任务进度状态（标记已完成的子主题）。

**⚠️ 禁止行为（违反任一条均须重做该子主题）**：
- **🚫 最严重违规**：未执行实际搜索，直接凭 AI 训练知识写 finding——**这是对研究质量的根本破坏**
- DO NOT 凭记忆构造 URL 直接抓取网页——**必须先用搜索引擎获取有效链接，再从搜索结果中选取 URL 通过 MCP 工具获取**
- DO NOT 同时启动超过 2 个子 Agent——**硬性并发上限为 2**
- DO NOT 在单个子 Agent 内同时发起超过 3 个并行搜索——**每子 Agent 搜索并发上限为 3**
- DO NOT 在委派子任务时使用开放性指令（如"缺口分析"、"补充搜索"、"自行判断"）——**主 Agent 必须预先决定所有搜索词和数据提取目标**
- DO NOT 以"任务繁多"、"快速整理"、"提高效率"为由跳过搜索步骤；子主题可以减少，但每个必须完整搜索
- DO NOT 在 findings 未全部完成时开始写 report.md
- DO NOT 仅读取 1 个 findings 就宣称“数据收集完成”或“可以直接开始写报告”
- DO NOT 未列出 `findings/` 目录的完整文件清单就开始正式撰写 `report.md` 正文
- DO NOT 把“后续会继续读取 findings”当作立即开始写报告的理由
- DO NOT 在仍有未消费 findings 时进入执行摘要、结论、HTML 或最终完成回复
- DO NOT finding 写得比报告章节还薄——finding 应该是报告的 1.5-2 倍信息量
- DO NOT 在搜索日志中填写虚构或无法访问的 URL

## 阶段 2.5：图片与视觉资产收集

**目标**：尝试收集 2-3 张与主题相关的图片 URL，用于丰富 HTML 报告的视觉呈现。**图片为可选增强项**，找不到合适图片不影响报告质量。

**前置条件**：所有子主题 findings 已完成

### 图片获取方式

**⚠️ 重要：你没有专门的"图片搜索"工具。** 普通的联网搜索工具（如 WebSearchPro、brave_search）只返回文本结果，不返回图片。

正确的图片获取方式是通过搜索工具找到图片页面 URL，然后通过网页抓取工具获取页面内容并提取图片 URL：

1. **Wikipedia 页面抓取**：用搜索工具找到主题的 Wikipedia 页面 URL → 通过网页抓取工具获取页面内容 → 从返回内容中提取 `https://upload.wikimedia.org/...` 格式的图片链接
2. **官方网站**：从研究阶段已访问过的权威来源页面中，回顾是否已获取到图片 URL

**关键约束**：
- **每章 = 一次独立的追加写入操作**，不可将多章合并到同一次操作
- **写入前必须先读取**：每章执行追加写入前，先读取 report.md 中对应章节的内容
- **逐段忠实转换**：report.md 中的每个段落 → HTML `<p>` 标签，在数据密集处穿插图表组件
- **不是重新概括，而是忠实转换格式** — 禁止对 report.md 内容进行缩写或摘要

## 质量标准

### 信息来源
- 每个重要论断至少引用 3 个独立来源
- 混合来源类型：官方文档、行业报告、专家分析、社区讨论
- 优先使用一手来源

### 报告深度
- **叙事优先**：每章以连续分析段落为主，表格仅作佐证
- 每章至少 6 段分析性段落（每段 150-250 字）
- 包含定量数据，为数字提供上下文（年份、来源、方法论）

### 视觉质量
- HTML 报告必须有 **Hero 标题区**（渐变背景）
- 至少 **4 个统计卡片**展示核心量化数据
- 至少 **2 种不同的图表组件**（条形图、环形图、进度条任选）
- **可选** 1-2 张嵌入图片（来自 Wikipedia Commons 等公开源，使用 `onerror` 防止图片404破坏布局）
- 报告末尾必须有**关键洞察 Takeaway 卡片**（3-5 条）

### 图片使用规范
- 只使用公开可直接访问的图片 URL
- 每张图片必须有说明文字（figcaption）
- 图片加载失败时自动隐藏（使用 `onerror="this.parentElement.style.display='none'"` 属性）
- 图片应与章节内容高度相关
