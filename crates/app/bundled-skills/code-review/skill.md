# 代码审查指令

当用户请求代码审查时，按以下框架执行。

## 审查维度（依次检查）

1. **逻辑正确性** — 算法是否对？边界条件？竞态条件？
2. **安全性** — OWASP Top 10（注入、XSS、认证绕过等）
3. **性能** — 不必要的循环、重复查询、内存泄漏、N+1 问题
4. **可维护性** — 命名、复杂度、重复代码、模块化
5. **错误处理** — 是否吞异常？是否暴露敏感信息？是否处理了所有失败路径？
6. **测试覆盖** — 关键路径有无测试？边界值？错误场景？
7. **兼容性** — API 变更是否向后兼容？数据库迁移可回滚？

## 输出格式

对每个文件按以下格式输出：

```
## 📁 src/xxx.ts

### 🔴 (严重) 问题描述
...
### 🟡 (建议) 问题描述
...
### ✅ 亮点
...
```

## 优先级

- 先处理关键路径和核心逻辑
- 再检查错误处理和边界情况
- 最后看代码风格和注释

## 规则

- 总是先读完整文件再评价，不能只看片段
- 不要为了找茬而找茬，真的有价值的问题才写
- 不要提代码风格偏好（除非项目有明确规范）
- 如果审查 diff，用 `git diff` 获取变更内容
- 严重问题需给出具体的修复建议和代码示例
