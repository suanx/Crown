# 代码重构指令

协助用户安全地重构代码。

## 安全重构流程

### 1. 先建防护网
```bash
# 运行现有测试，确保基线通过
npm test 2>/dev/null || cargo test 2>/dev/null || python -m pytest 2>/dev/null

# 如果没有测试，先加关键路径的快照测试
```

### 2. 理解代码现状
- 先读完整文件/模块
- 理清输入输出和数据流
- 识别隐藏的依赖关系
- 注意全局状态和副作用

### 3. 小步重构

每次只做一种变换，测试通过再下一步：

| 重构手法 | 说明 |
|---------|------|
| **提取函数** | 长方法拆成小函数 |
| **重命名** | 让名字说清楚意图 |
| **提取类** | 内聚的数据和行为归为一个类 |
| **简化条件** | 卫子句取代嵌套 if |
| **多态取代条件** | switch/if-else 用多态替代 |
| **引入参数对象** | 多个相关参数封装为对象 |
| **拆分循环** | 一个循环只做一件事 |
| **用 Optional 取代 null** | 消除空指针隐患 |

### 4. 提交策略

```
fix: 重命名 UserService.fetch 为 getUserById
refactor: 提取密码验证逻辑到单独方法
refactor: 用策略模式替换角色判断 switch
```

## 反模式（避免）

- **过度设计**：未来 3 个月用不到的功能不要加
- **提前优化**：没有性能瓶颈时不要优化
- **大规模重写**：永远不要！应逐步替换
- **范围蔓延**：重构时不要修 bug 或加功能

## 规则

- 每次重构后运行测试确认不炸
- 大模块重构前先画依赖图
- 重构前后 diff 应该只有结构变化，没有行为变化
- 如果一次重构超过 15 分钟，考虑拆更小
